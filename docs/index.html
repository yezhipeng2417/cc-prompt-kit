Techniques

8 Core Techniques

8 项核心技巧

Each technique maps to a specific pattern found in Claude Code's production prompts.

每项技巧都对应 Claude Code 生产级 prompt 中的具体模式。

01

Identity Framing

身份定义

Define a precise role, strengths, and boundaries in the first 1-3 sentences. The identity constrains behavior implicitly — "code review specialist" means "you don't write frontend code" without saying it.

在前 1-3 句中定义精确角色、优势和边界。身份隐式约束行为——"代码审查专家"意味着"你不写前端代码"而无需明说。

You are a helpful assistant that reviews code.
You are a code review specialist. Your strengths are catching security vulnerabilities and performance issues. You NEVER suggest stylistic changes unless they affect correctness.
Source: Every agent in src/tools/AgentTool/built-in/
来源：src/tools/AgentTool/built-in/ 中的所有 Agent
02

Negative Constraints + Alternatives

负面约束 + 正面替代

Every NEVER needs a "do this instead." A bare prohibition creates a void. The model knows what NOT to do but not what TO do, leading to awkward workarounds.

每条 NEVER 都需要"替代方案"。裸禁令制造真空——模型知道不该做什么，但不知道该做什么，导致笨拙的变通。

Don't make up information.
NEVER fabricate citations. If you don't have the data, say "I don't have this information" and suggest where to find it.
Source: Git safety protocol in BashTool/prompt.ts
来源：BashTool/prompt.ts 中的 Git 安全协议
03

Contrastive Examples

正反对比示例

One RIGHT/WRONG pair is worth ten sentences of instruction. The model extracts the quality standard from the contrast, not from adjectives like "concise" or "professional."

一组正/反示例胜过十句描述。模型从对比中提取质量标准，而不是从"简洁""专业"等形容词中。

Write concise commit messages.
Write concise commit messages. RIGHT: "Fix null pointer in auth when session expires" WRONG: "Fixed a bug that was causing issues with the authentication system when users had expired sessions"
Source: BashTool description parameter guidance
来源：BashTool 的 description 参数指南
04

Anti-Pattern Naming

失败模式命名

Give failure modes memorable names. Once a concept has a label ("Verification Theater"), the model treats it as something to actively avoid rather than a vague warning.

给失败模式起记忆性强的名字。概念一旦有了标签（"验证剧场"），模型就会主动回避，而非仅仅注意一个模糊警告。

Don't just read the code and say it looks fine.
Avoid "Verification Theater" — the failure mode where you read code and declare it correct without running it. Real verification requires executing commands and checking output.
Source: verificationAgent.ts lines 10-130
来源：verificationAgent.ts 第 10-130 行
05

Process Templates

流程模板

Numbered steps prevent reordering and skipping. Prose-described workflows are ambiguous about ordering and optionality. The model can't distinguish mandatory steps from suggestions.

编号步骤防止跳过和乱序。散文描述的流程对顺序和可选性含糊不清，模型无法区分必需步骤和建议。

Analyze the security of this code.
Follow this sequence: 1. SCAN — read all modified files 2. PROFILE — identify vulnerability patterns 3. VALIDATE — check against OWASP top 10 4. REPORT — structured findings per template Do not skip to step 4.
Source: Plan Agent's 4-phase process, Compact prompt's 9-section template
来源：Plan Agent 四阶段流程，Compact prompt 九段式模板
06

Graduated Specificity

分级详细度

Match instruction detail to complexity. Give the model permission to be brief on simple cases and show it what "detailed" means on complex ones.

让指令详细度匹配复杂度。允许模型在简单情况下简短回答，同时展示复杂情况下"详细"是什么样。

Write clear descriptions for all commands.
Simple commands (5-10 words): ls → "List files in current directory" Complex commands (add context): curl -s url | jq '.data[]' → "Fetch JSON and extract data array"
Source: BashTool description levels, Explore Agent's "quick"/"medium"/"very thorough"
来源：BashTool 描述分级，Explore Agent 的 "quick"/"medium"/"very thorough"
07

Economic Reasoning

经济推理

"This is faster/cheaper" persuades better than "you must." Explaining why an approach is efficient works better than mandating it.

"这样更快/更省"比"你必须"更有说服力。解释为什么某种方法更高效，比命令式更有效。

Always use the Grep tool instead of bash grep.
The Grep tool is optimized for this environment — faster than bash grep and handles permissions correctly. Use it for all search tasks.
Source: Agent Tool — "Forks are cheap, they share the parent's prompt cache"
来源：Agent Tool——"Fork 很便宜，它们共享父级的 prompt 缓存"
08

Cache-Aware Structure & Defense in Depth

缓存结构与防御纵深

Static content first, dynamic last, separated by a boundary for prompt caching. Critical constraints appear in the constraints section AND inline at the point of action — different prompt positions get different attention weight.

静态内容在前，动态在后，用边界分隔以支持缓存。关键约束在约束区和行动点都出现——长 prompt 的不同位置获得不同的注意力权重。

[Mixed: instructions, context, more instructions, context...]
[STATIC — cached] Identity, constraints, process, examples — CACHE BOUNDARY — [DYNAMIC — per request] User context, session state, data
Source: SYSTEM_PROMPT_DYNAMIC_BOUNDARY in prompts.ts line 114
来源：prompts.ts 第 114 行的 SYSTEM_PROMPT_DYNAMIC_BOUNDARY
Insights

3 Architectural Insights

3 个架构级洞察

Higher-level principles that emerge from the source beyond individual techniques.

超越单项技巧的高层设计原则。

1

Prompts as API Contracts

Prompt 是 API 契约

Claude Code's prompts read like interface docs — identity as type definition, MUST/NEVER as contracts, examples as test cases, output templates as return types. Write prompts like specs, not emails.

Claude Code 的 prompt 读起来像接口文档——身份是类型定义，MUST/NEVER 是契约，示例是测试用例，输出模板是返回值类型。像写规格说明一样写 prompt，而不是写邮件。

2

Defense in Depth

防御纵深

The same rule ("don't force push") appears in the system prompt, in BashTool's prompt, AND in code-level validation. Critical rules get stated twice — once in constraints, once at the action point. Never trust one layer.

同一条规则（"不要 force push"）在系统提示、BashTool 提示和代码层面各出现一次。关键规则声明两次——约束区一次，行动点一次。永远不要信任单层防线。

3

Separation of Concerns

关注点分离

Context (git status, memory), Capability (tool definitions), and Behavior (instructions) are injected via separate paths. Don't mix what the AI knows, what it can do, and how it should behave.

上下文（git 状态、记忆）、能力（工具定义）、行为（指令）通过独立路径注入。不要混淆 AI 知道什么、能做什么、应该怎么做。
