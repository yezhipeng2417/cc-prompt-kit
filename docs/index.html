Techniques

8 Core Prompt Engineering Techniques

Every technique traces back to a specific pattern found in Claude Code's production prompts.

01

Identity Framing · 身份定义

A prompt without a clear identity produces generic output. The identity isn't decoration — it defines the capability boundary. Claude Code's every agent opens with a precise role + strengths + constraints frame.

You are a helpful assistant that reviews code.
You are a code review specialist. Your strengths are catching security vulnerabilities and performance issues. You NEVER suggest stylistic changes unless they affect correctness.
Source: Every agent prompt (Explore, Plan, Verification) in src/tools/AgentTool/built-in/
02

Negative Constraints + Alternatives · 负面约束+正面替代

Every NEVER rule in Claude Code has a corresponding positive action. A bare prohibition creates a void — the model knows what NOT to do but not what TO do. The fix: pair every prohibition with "instead, do Y."

Don't make up information.
NEVER fabricate citations or statistics. If you don't have the data, say "I don't have this information" and suggest where to find it.
Source: Git safety protocol in src/tools/BashTool/prompt.ts — "NEVER skip hooks. If a hook fails, investigate and fix the underlying issue."
03

Contrastive Examples · 正反对比示例

One good RIGHT/WRONG example pair is worth ten sentences of instruction. The model extracts the implicit quality standard from the contrast. Claude Code uses this pattern in every tool description and agent prompt.

Write concise commit messages.
Write concise commit messages.

RIGHT: "Fix null pointer in auth when session expires"
WRONG: "Fixed a bug that was causing issues with the authentication system when users had expired sessions"
Source: BashTool description parameter guidance with concrete command examples
04

Anti-Pattern Naming · 失败模式命名

Naming a failure mode makes it a concept the model actively avoids. Claude Code's Verification Agent names two patterns: "Verification Avoidance" (reading code instead of running it) and "Seduced by 80%" (declaring pass after first checks).

Don't just read the code and say it looks fine — actually run it.
Avoid "Verification Theater" — the failure mode where you read code and declare it correct without actually executing it. Real verification requires running commands and checking output.
Source: src/tools/AgentTool/built-in/verificationAgent.ts lines 10-130
05

Process Templates · 流程模板

For tasks requiring consistent output, provide a fill-in-the-blank structure. Claude Code's Plan Agent uses a rigid 4-phase process; the Compact prompt has a 9-section summary template. Numbered steps prevent reordering and skipping.

Analyze the security of this code.
Follow this structure:
1. SCAN: Read all modified files
2. PROFILE: Identify patterns
3. VALIDATE: Check against rules
4. REPORT: Structured findings

Do not skip to step 4.
Source: Plan Agent 4-phase process, Compact prompt 9-section template, Commit command workflow
06

Graduated Specificity · 分级详细度

Don't use the same level of detail for simple and complex cases. Claude Code's BashTool gives brief guidance for simple commands and detailed guidance for complex ones. The Explore Agent accepts three thoroughness levels.

Write clear descriptions for all commands.
Simple commands (5-10 words):
• ls → "List files in current directory"

Complex commands (add context):
• curl -s url | jq '.data[]' → "Fetch JSON and extract data array"
Source: BashTool description parameter, Explore Agent thoroughness levels ("quick" / "medium" / "very thorough")
07

Economic Reasoning · 经济推理

"This approach is faster/cheaper" persuades better than "you must." Claude Code's Agent prompt explains "Forks are CHEAP — they share the parent's prompt cache" rather than mandating fork usage.

Always use the Grep tool instead of bash grep.
The Grep tool is optimized for this environment — it's faster than bash grep and handles permissions correctly. Use it for all search tasks.
Source: Agent Tool prompt — "Don't set model on fork — different model can't reuse parent cache"
08

Cache-Aware Structure & Defense in Depth · 缓存结构 & 防御纵深

Static content first, dynamic content last, separated by a clear boundary — for prompt caching cost savings. Critical constraints appear both in the constraints section AND inline at the point of action. This redundancy is intentional: different positions in a long prompt receive different attention weight.

[Mixed: instructions, context, more instructions, more context...]
[STATIC — cached]
Identity, constraints, process, examples

— CACHE BOUNDARY —

[DYNAMIC — per request]
User context, session state, current data
Source: SYSTEM_PROMPT_DYNAMIC_BOUNDARY in src/constants/prompts.ts line 114
Insights

3 Architectural Insights

Beyond individual techniques, three higher-level design principles emerge from the source.

1

Prompts Are API Contracts
Prompt 是 API 契约

Claude Code's prompts read like interface documentation — identity as type definition, MUST/NEVER as contracts, examples as test cases, output templates as return types. Write prompts like specifications, not emails.

2

Defense in Depth
Prompt 层的防御纵深

The same constraint ("don't force push") appears in the system prompt, in BashTool's prompt, AND in code-level validation. Critical rules get stated twice — once in constraints, once at the point of action. Never trust a single layer.

3

Separation of Concerns
关注点分离

Claude Code separates Context (git status, memory), Capability (tool definitions), and Behavior (system prompt) into independent injection paths. Don't mix "what the AI knows" with "what it can do" with "how it should behave."
